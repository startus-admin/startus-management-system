# STARTUS 運用ロードマップ

作成日: 2026年3月13日

---

## 現状の課題

```
現在の業務フロー（人為的なデータ受け渡しが多い）

  保護者 → Webフォーム → メール通知 → 事務局が手動で確認
                                         ↓
                          SharePoint / 紙 / スグラムに手動転記
                                         ↓
                          出席簿に手動反映 / 指導者に口頭連絡
```

**問題点:**
- 同じデータを複数のシステムに手動で入力 → **転記ミス・漏れ**
- SharePoint、スグラム、メール、紙が混在 → **情報が散在**
- 誰がどこまで処理したか不明確 → **対応漏れ**
- データが各所にバラバラ → **分析・活用ができない**

---

## 目指す姿

```
目標の業務フロー（データが自動で流れる）

  保護者 → Webフォーム → Supabase（自動保存）
                              ↓ 自動反映
                    STARTUS管理アプリで一括管理
                     ├→ 申請処理（承認→会員ステータス自動更新）
                     ├→ 出欠アプリ（会員データ自動連携）
                     ├→ チャット（スタッフ間連絡）
                     └→ 将来：独自コミュニケーションアプリ（保護者連絡）
```

## ステップ1: Googleアカウント認証の整備

### やること

| 項目 | 内容 |
|------|------|
| 全職員にGoogleアカウントを付与 | `@startus-kanazawa.org` or 個人Gmail |
| 二段階認証（2FA）を有効化 | Google Authenticator or SMS認証 |
| PCログインをGoogleアカウントで統一 | Chromeプロファイルでログイン管理 |

### なぜこれが最初か

- STARTUS管理アプリは **Google OAuth認証** でログインする仕組み
- 2FAが有効でないと、パスワード漏洩時にシステム全体が危険
- **これが全ての土台** → アプリを使うにはまずここから

### 完了条件
- [ ] 全職員がGoogleアカウントを持っている
- [ ] 全員が2FA（二段階認証）を有効にしている
- [ ] PCでGoogleにログインし、管理アプリにアクセスできる
- [ ] 管理アプリの `ALLOWED_EMAILS` に全員を登録済み

## ステップ2: SharePoint Listsの使用停止

### 現状

```
フォーム送信時のデータ保存先:
  ① Supabase（管理アプリのDB） ← 新しい（既に稼働中）
  ② SharePoint Lists           ← 古い（バックアップ用に残存）
  ③ メール通知                  ← 確認用
```

### やること

| 項目 | 内容 |
|------|------|
| SharePointへの書込みを停止 | GAS側の `saveToSharePoint()` を無効化 |
| 過去データの保全 | 必要分をCSVエクスポートして保管 |
| Azure AD連携の廃止 | CLIENT_SECRET等の認証情報を削除 |

### メリット

- **シンプル化**: データ保存先が Supabase 1箇所に → 「どっちが正しい？」がなくなる
- **セキュリティ向上**: Azure AD の複雑な認証設定（CLIENT_SECRET, Tenant ID等）が不要に
- **保守コスト削減**: SharePoint側の権限管理・トラブル対応が不要に

### 完了条件
- [ ] 全フォームがSupabaseのみにデータ保存している
- [ ] SharePointへの新規書込みが完全に停止している
- [ ] 過去データのバックアップが完了している
- [ ] Azure AD関連の認証情報がコードから削除されている

## ステップ3: STARTUS管理アプリの本格運用

### 使える機能（既に実装済み）

| 機能 | できること | 置き換え対象 |
|------|-----------|-------------|
| **申請管理** | フォームからの申請を一覧表示・承認・却下 | メール確認→手動処理 |
| **会員管理** | 会員情報の検索・編集・ステータス管理 | SharePoint / 紙の名簿 |
| **教室管理** | 教室・サブクラスの設定 | 手動管理 |
| **出欠管理** | 出欠アプリとデータ連携（出欠記録の確認） | 紙の出席簿 |
| **スタッフ管理** | 職員情報・権限管理 | 口頭 / メール |
| **チャット** | スタッフ間メッセージ・ファイル共有 | LINE / メール |
| **月謝管理** | 月謝の確認・未納チェック | 手動チェック |
| **ダッシュボード** | 未処理件数の通知バッジ | 確認漏れ |

### 業務フローの変化（例：入会申請）

```
【Before】
  保護者がフォーム送信
    → メールで通知が届く
    → 事務局がメールを見て確認
    → SharePointに転記
    → 出席簿に手動追加
    → 指導者にLINEやスグラムで連絡
    → 月謝台帳に手動追加

【After】
  保護者がフォーム送信
    → Supabaseに自動保存 → 管理アプリに即反映（通知バッジ）
    → 事務局がアプリで「承認」ボタン → 会員ステータス自動更新
    → 出欠アプリに自動反映
    → チャットで指導者に連絡
    → 月謝管理に自動反映
```

**手動作業: 6ステップ → 2ステップ（承認＋連絡）に削減**

### 完了条件
- [ ] 事務局スタッフが管理アプリにログインして日常業務を実施
- [ ] 新規申請は管理アプリで処理（メール確認→手動転記を廃止）
- [ ] 出欠アプリを現場で利用開始

## ステップ4: スグラム → 独自コミュニケーションアプリ（将来）

### 現状のスグラム

スグラム（Sgrum）= スポーツクラブ向けコミュニケーションアプリ
- 保護者への連絡・お知らせ配信
- 出欠連絡の受付
- 写真共有

### 独自アプリに置き換える理由

| 観点 | スグラム（外部） | 独自アプリ |
|------|----------------|-----------|
| **データ連携** | 手動でデータを転記 | Supabaseと直結→自動連携 |
| **カスタマイズ** | スグラム社の仕様に依存 | 業務に合わせて自由に設計 |
| **コスト** | 月額利用料が発生 | 自社インフラ（無料枠内） |
| **データ所有権** | スグラム社のサーバーに保管 | 自社Supabaseに保管 |
| **機能拡張** | 要望しても対応は不明 | 必要な機能を随時追加 |

### 実現イメージ

```
独自コミュニケーションアプリ（将来構想）:
  ├→ 保護者向け機能
  │    ├ お知らせ配信（プッシュ通知）
  │    ├ 欠席・振替の申請（→ 管理アプリに自動連携）
  │    ├ 月謝・支払い状況の確認
  │    └ 教室スケジュールの確認
  │
  ├→ 指導者向け機能
  │    ├ 本日の出欠状況確認
  │    ├ 教室ごとの会員リスト
  │    └ スタッフ間チャット（既存機能を拡張）
  │
  └→ データ連携（すべて自動）
       ├ 会員マスター ← Supabase
       ├ 出欠データ ← 出欠アプリ
       ├ 申請データ ← 管理アプリ
       └ 月謝データ ← 管理アプリ
```

## 全体のゴール：データ駆動型のクラブ運営

### データの一元管理がもたらすもの

```
┌──────────────────────────────────────────────────┐
│              Supabase（単一データベース）              │
│                                                  │
│  会員データ ─── 申請データ ─── 出欠データ              │
│      │              │              │              │
│  月謝データ ─── スタッフ ──── 教室データ              │
│      │              │              │              │
│      └──────────┬───┘──────────────┘              │
│                 │                                │
│           ┌─────┴─────┐                          │
│           │  AI分析    │  ← 将来Phase            │
│           └─────┬─────┘                          │
│                 │                                │
│    会員動向 / 月謝状況 / 教室稼働率 / 転換率         │
└──────────────────────────────────────────────────┘
```

### 3つの価値

#### 1. ミスの排除
| Before | After |
|--------|-------|
| 人がデータを手で転記 | システムが自動連携 |
| 「入力忘れ」「転記ミス」が常態化 | データは1回入力→全体に反映 |
| 「あの申請どうなった？」が頻発 | ステータスがリアルタイムで可視化 |

#### 2. データをビジネスの強みに
| 活用例 | 内容 |
|--------|------|
| **体験→入会の転換率** | どの教室・時期の転換率が高いか分析 → 効果的な集客 |
| **退会予兆の検出** | 出欠率の低下パターンを検知 → 早期フォロー |
| **教室稼働率の最適化** | 各教室の定員充足率を可視化 → 新教室の開設判断 |
| **月謝の徴収改善** | 未納パターンの分析 → 自動リマインド |

#### 3. シンプル＆安全な管理
| Before | After |
|--------|-------|
| SharePoint + スグラム + メール + 紙 | **Supabase 1箇所** |
| 各システムのパスワード管理が煩雑 | **Googleアカウント1つ + 2FA** |
| データがどこにあるか分からない | **1つのDBに全データ集約** |
| 退職者のアクセス権剥奪が漏れがち | **Googleアカウント無効化で即遮断** |
